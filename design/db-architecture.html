4b. Conditional Logic (Spans Form Structure + Registry)

Scenario: On the Employee Onboarding form, when Department = “Engineering”, show a “Tech Stack” field and make it required. Each condition group targets one element, has an action, and contains one or more rules combined with AND/OR logic.

form_element_conditions — Condition groups: “When these rules are met, do this action”
idform_element_idaction_type_idaction_valuelogic_operatordisplay_order
1 5 (tech_stack) 1 (show) NULL AND 0
2 5 (tech_stack) 3 (require) NULL AND 1

Two separate condition groups on the same element: one to show it, one to make it required. Both trigger when dept = engineering.

form_element_condition_rules — Individual rules within a condition group
idcondition_idsource_element_idoperator_idvaluedisplay_order
1 1 (show tech_stack) 4 (department) 1 (equals) eng 0
2 2 (require tech_stack) 4 (department) 1 (equals) eng 0

Each rule says: “check [source_element] [operator] [value]”. Multiple rules in a group are combined by the group's logic_operator (AND/OR).

CONDITION LOGIC VISUALIZATION
tech_stack field:
  Condition 1 (action: SHOW):
    WHEN department EQUALS "eng"    → show this field
  Condition 2 (action: REQUIRE):
    WHEN department EQUALS "eng"    → make this field required

At runtime, the form renderer evaluates these conditions
as the user fills the form, showing/hiding and toggling
validation in real-time.

7. Key Queries the App Would Run

Load left sidebar (element palette)
SELECT et.id, et.name, et.label, et.icon, et.is_layout,
       ec.label as category_label, ec.name as category_name
FROM element_types et
JOIN element_categories ec ON et.category_id = ec.id
ORDER BY ec.display_order, et.display_order;
Load right sidebar when clicking a textfield
SELECT pd.id, pd.name, pd.label, pd.data_type, pd.input_type,
       pd.default_value, pg.label as group_label, pg.name as group_name,
       etp.is_required, etp.override_default, etp.display_order
FROM element_type_properties etp
JOIN property_definitions pd ON etp.property_definition_id = pd.id
JOIN property_groups pg ON pd.property_group_id = pg.id
WHERE etp.element_type_id = ?  -- the clicked element's type
ORDER BY pg.display_order, etp.display_order;
Reconstruct full form for rendering
SELECT fe.id, fe.element_key, fe.position, fe.parent_id,
       et.name as type, et.is_layout,
       fev.value as prop_value, pd.name as prop_name, pd.data_type
FROM form_elements fe
JOIN element_types et ON fe.element_type_id = et.id
LEFT JOIN form_element_values fev ON fev.form_element_id = fe.id
LEFT JOIN property_definitions pd ON fev.property_definition_id = pd.id
WHERE fe.form_version_id = ?
ORDER BY fe.parent_id NULLS FIRST, fe.position;
+ separate query for options: SELECT * FROM form_element_options WHERE form_element_id IN (...)
Load conditions for form elements (for rendering + builder)
SELECT fec.id as condition_id, fec.form_element_id as target_id,
       cat.name as action, fec.action_value, fec.logic_operator,
       fecr.source_element_id, fe_src.element_key as source_key,
       co.name as operator, fecr.value
FROM form_element_conditions fec
JOIN condition_action_types cat ON fec.action_type_id = cat.id
JOIN form_element_condition_rules fecr ON fecr.condition_id = fec.id
JOIN condition_operators co ON fecr.operator_id = co.id
JOIN form_elements fe_src ON fecr.source_element_id = fe_src.id
WHERE fec.form_element_id IN (
  SELECT id FROM form_elements WHERE form_version_id = ?
)
ORDER BY fec.form_element_id, fec.display_order, fecr.display_order;
